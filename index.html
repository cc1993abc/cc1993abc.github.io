design-pattern principal

发表于 2021-05-03

设计模式的六大原则

依赖倒置原则

高层模块不应该依赖低层模块，两者都应该依赖其抽象；抽象不应该依赖细节，细节应该依赖抽象。

个人理解补充:面向接口变成,比如Controller应使用封装好的接口,而由具体的Service去实现Iservice,面向接口变成就是依赖倒置的一种体现,也避免了循环依赖。 接口也随时可以被重写或者拆分

单一职责原则SRP

官方翻译：应该有且仅有一个原因引起类的变更。简单点说，一个类，最好只负责一件事，只有一个引起它变化的原因。

个人理解:比如一个搞艺术的人,我们应具体把他拆分为谈钢琴,画画 对艺术这件事做拆分。 拆分细致程度应在系统中被上层引用时不可再拆位置,这样避免改动时影响系统的关联关系。这和DDD中的一些思想类似

接口隔离原则ISP

其一是不应该强行要求客户端依赖于它们不用的接口；其二是类之间的依赖应该建立在最小的接口上面。简单点说，客户端需要什么功能，就提供什么接口，对于客户端不需要的接口不应该强行要求其依赖；类之间的依赖应该建立在最小的接口上面，这里最小的粒度取决于单一职责原则的划分。

个人理解:这个与单一职责很类似,单一职责强调的是业务拆分,接口隔离是设计架构的约束,比如加密算法不应和HTTP请求的扩展放在一起

开闭原则OCP

Software entities (classes, modules, functions) should be open for extension but closed for modification。（这个英文字面意思更好）

个人理解:尽可能的不改变原有代码的情况下去做扩展,因为对原有的代码进行扩展,一个小的比方,那么使用了他的代码都需要重新审查一遍。比如微软的WebClient技术和HTTPClient技术是开新模块去写的,不可能说在WebClient的基础上去拓展HTTP请求方法

里氏转换原则

里氏代换原则(Liskov Substitution Principle, LSP)：所有引用基类（父类）的地方必须能透明地使用其子类的对象。

原则：子类可以扩展父类的功能，但不能改变父类原有的功能。

父类能出现的地方都可以用子类来代替，而且换成子类也不会出现任何错误或异常，而使用者也无需知道是父类还是子类，

但反过来则不成立。总之，就是抽象。

  1. 子类必须完全实现父类的抽象方法，但不能覆盖父类的非抽象方法；

  2. 子类中可以增加自己特有的方法；

  3. 当子类的方法重载父类的方法时，方法的前置条件(即方法的形参)要比父类方法的输入参数要更宽松；

    4.当子类的方法实现父类的抽象方法时，方法的后置条件(即方法的返回值)要比父类更严格。

个人理解:这个在我们比较快速的开发代码迭代中确实很难做到,因为一般的业务中,你很难从一开始就是抽象出一些方法的业务上的共性,基础工具类使用可以尽量保证里氏替换原则,提高基础工具类的重用性。比如我们可以抽象出一个logger基类, 在微服务项目中使用简单的日志生成,后续替换为分布式日志

迪米特法则

一个软件实体应当尽可能少地与其他实体发生相互作用。

个人理解:有时候功能设计,我们可以通过一个中介(例如kafka)去作为中介,去通信生产方和消费方,比如登录功能时 发送短信, 验证码功能,我们可以通过kafka,而不是前端输入账号 + 发送验证码 + 验证验证码相互传导。

合成复用原则

合成复用原则（Composite Reuse Principle, CRP）就是在一个新的对象里面使用一些已有的对象，使之成为新对象的一部分。新对象通过向这些对象的委派达到复用已用功能的目的。简单地说，就是要尽量使用合成/聚合，尽量不要使用继承。

发表于 2020-08-09

HTTP

HTTP 1.0 VS HTTP 2.0

  • 效率上的提升，减少连接次数，压缩请求头部及尾部
  • 双工通信，服务器推送

    HTTPS VS HTTP

  • HTTPS采用混合加密方式，对称加密保证传输效率，非对称加密保证安全性。
  • HTTPS需要进行解密加密等过程，(SSL协议提供了报文的完整性保护)相比HTTP速度会更慢，而且需要支付一定的费用。
